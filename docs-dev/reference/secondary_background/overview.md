# roast-a-researcher — planning overview

*Status: design settled; not yet built. The next deliverable is a binding `specification.md`. This document records the decisions taken and the points still open.*

---

## Concept

A browser-based tool that takes a researcher's own profile information and generates a comedic "roast" of it rather than a sincere academic profile. It is a self-service generator: any researcher runs it on their own material, and the output is produced per user from supplied input, not a hand-written page about one person.

The sincere counterpart to this idea already exists as ScholarFolio (scholarfolio.org). That project is treated here as inspiration and a worked example, not as a base to fork. The reasons are recorded under "Relationship to ScholarFolio" below.

---

## Decisions taken

- The project name is `roast-a-researcher`.
- It is a self-service generator for any researcher, not a profile of one person.
- The front end is a static site (Vite + TypeScript) hosted on GitHub Pages.
- It is a fresh repository generated from the `eek-a-template` Copier template (stack `node`, static site, public), not a fork of ScholarFolio or of the existing personal website.
- Roast text is generated by a large language model via OpenRouter.
- The model is called through a small backend (a Cloudflare Worker) that holds a single OpenRouter key belonging to the project owner. Users do not supply a key.
- Spending is controlled at two levels: a per-user rate limit enforced in the Worker, and a hard spend cap on the OpenRouter account.
- Input for the first version is a single paste field. The user pastes their own material (CV, publication list, bio, an exported profile). The tool does not scrape third-party sites.
- The build follows the eek-a-dev workflow: the design is fixed in `specification.md`, and Claude Code implements it through numbered prompts, one commit at a time.

Points still open are listed under "Open decisions".

## Architecture

### Front end and hosting

GitHub Pages serves static files only; it cannot run server-side code or hold a secret. The interface (HTML, CSS, JavaScript) is therefore entirely static and is generated and deployed through the template's existing GitHub Pages workflow. The single capability a static host lacks, a place to keep the API key, is supplied by the backend below rather than by pushing the key into the browser.

### Backend: a key-holding proxy

The first version of this plan assumed a bring-your-own-key (BYOK) model, in which each user supplied their own key so that no secret needed to live on the server. That has been replaced. The owner now provides one key, so that the tool works for any visitor without setup, and a small backend is required to hold that key out of the browser.

The backend is a Cloudflare Worker. Its responsibilities are narrow:

- hold the OpenRouter key as a secret, never exposed to the client;
- receive a request from the static page (the pasted text plus the chosen model);
- forward it to OpenRouter with the key attached, and return the result;
- enforce a per-user rate limit before forwarding, so no single visitor or bot can exhaust the budget.

A public static key would be readable by every visitor and quickly drained; the Worker exists specifically to prevent that. The rate-limit state (a per-IP counter with a daily reset) is the only thing the Worker stores; it uses a Cloudflare key-value store or the platform's rate-limiting facility for this. The Worker holds no profile data after a request completes.

### Input

The first version collects input through one paste field. A deployed static page cannot fetch a user's Google Scholar, LinkedIn, or ResearchGate profile directly (cross-origin restrictions and authentication), so the user supplies the text. The field accepts anything: a CV, a publication list, an exported profile page, a bio.

Short helper lines in the interface explain how to obtain that text, for example:

- **LinkedIn:** open your profile, use the **Save to PDF** option in the More/Resources menu beneath your photo, then paste the text from that PDF.
- **Scholar Folio / Google Scholar:** open your public profile page and copy its text.
- **ORCID:** paste your ORCID iD or the contents of your public record.

Automated retrieval is deliberately deferred to later versions, where the Worker makes it feasible without scraping. The realistic, low-friction sources, in order of value for this audience:

- **ORCID** — the only major academic source with an official public API; from an ORCID iD the Worker can fetch public works and affiliations.
- **GitHub** — an official, free API; useful material for technically active researchers.
- **OpenAlex / Crossref / Semantic Scholar** — free open APIs for publications, co-authors, and citation metrics, keyed by name, DOI, or ORCID.
- **A URL fetcher** — the Worker retrieves the text of a public page (personal site, staff page, Scholar profile) the user points to.

These are recorded as planned enhancements, not first-version scope.

### Generation and key handling

Roast text is produced by a single LLM call. The flow is: the browser posts the pasted text to the Worker; the Worker attaches the key and calls OpenRouter; the response returns to the browser. The key is never present in the static site.

The BYOK alternatives from the earlier plan (a user pasting their own key, or the OpenRouter PKCE one-click connect flow) are no longer part of the first version, because users no longer supply keys. They remain viable if an owner-funded model is ever undesirable: PKCE in particular would let each user spend from their own account. They are retained here only as a recorded alternative.

A fully static, no-LLM fallback also remains possible: a deterministic template engine could assemble a roast from the pasted text with no API call and no data leaving the browser. This is not the chosen direction but is noted as a lower-friction, zero-cost alternative and a possible offline mode.

## Provider selection: OpenRouter

OpenRouter is retained as the provider, now called from the Worker rather than the browser:

- one key reaches a large catalogue of models from multiple providers through a single endpoint;
- the endpoint is OpenAI-compatible, so the integration is written once;
- model fallback routing is available if a chosen model is unavailable.

### Integration essentials

- Endpoint: `https://openrouter.ai/api/v1/chat/completions`, OpenAI-compatible, authenticated with `Authorization: Bearer <key>` (the key supplied by the Worker, not the client).
- Request body: a `model` slug plus a `messages` array (system prompt and the pasted profile text).
- Model slugs come from `openrouter.ai/models`; a "latest" alias lets the code follow newer versions without redeployment. A small or flash-class model is sufficient.
- Streaming is available with `stream: true` (Server-Sent Events) and is the natural way to show the roast as it is produced.

### Call shape

The browser calls the Worker; the Worker calls OpenRouter.

```js
// In the browser: call the project's own Worker, never OpenRouter directly.
const res = await fetch("https://roast-a-researcher.<account>.workers.dev/roast", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ model: "<slug>", profile: pastedProfileText }),
});
const { roast } = await res.json();
```

```js
// In the Worker: attach the secret key and forward to OpenRouter.
const upstream = await fetch("https://openrouter.ai/api/v1/chat/completions", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${env.OPENROUTER_KEY}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    model,
    messages: [
      { role: "system", content: ROAST_SYSTEM_PROMPT },
      { role: "user", content: profile },
    ],
  }),
});
```

## Spending controls and abuse protection

Because the owner funds usage, cost control is a first-version requirement rather than an afterthought. Two layers apply:

- **Per-user rate limit (Worker).** A daily cap on roasts per IP, enforced before any upstream call. This prevents a single visitor or a bot from consuming the whole budget and keeps the allowance shared fairly.
- **Account spend cap (OpenRouter).** A hard credit limit on the key, as an absolute backstop independent of the Worker.

Per-roast cost is small. The estimate below assumes roughly 5,000 input tokens (instructions plus a pasted CV or publication list) and 1,000 output tokens; longer input scales the input portion linearly. The table is a reference for choosing which model the Worker routes to, not a BYOK comparison; under the owner-funded model the project's cost is no longer zero.

| Provider / model | Input / output (per 1M tokens) | ≈ cost per roast | Notes |
|---|---|---|---|
| Mistral Small 3.1 | $0.20 / $0.60 | ~$0.0016 | EU-hosted; Large 3 at $2 / $6 |
| Google Gemini 2.5 Flash | $0.30 / $2.50 | ~$0.004 | Generous free tier; 2.0 Flash cheaper |
| DeepSeek V4 Flash | $0.14 / $0.28 | ~$0.001 | See privacy note on residency |
| Groq (Llama 3.1 8B) | $0.05 / $0.08 | ~$0.0004 | Open-weight models only |
| xAI Grok 3 Mini | $0.30 / $0.50 | ~$0.002 | |
| Claude Haiku 4.5 | $1 / $5 | ~$0.01 | Higher prose quality |
| OpenAI GPT-5.4 | $2.50 / $15 | ~$0.03 | Nano tier far cheaper |

At this volume token price is effectively a non-differentiator; the rate limit and spend cap, not the per-token rate, are what bound the owner's exposure. A flash-class model keeps the marginal cost negligible while leaving prose quality adequate for comedy.

## Relationship to ScholarFolio

ScholarFolio (jheller1212/ScholarFolio, MIT-licensed) is a sincere research-portfolio tool: from a Google Scholar URL it assembles publication history, a co-authorship network, citation trends, and impact metrics. It is a useful example of what researcher data can be turned into, and its MIT licence permits reuse of specific parts.

It is not forked, for two reasons. First, its goal differs: it is a sincere, data-rich product with accounts, a credit system, and Stripe payments, built on Supabase Edge Functions, a Postgres database, and SerpAPI for Scholar scraping. Adopting it would mean inheriting and then stripping that infrastructure to fit a small comedic tool. Second, the priority for this project is a small codebase that is fully owned and understood, which a fresh build serves better than adapting a larger one.

If richer data is wanted later, the intended approach is to reimplement specific, well-scoped pieces (for example, Scholar-profile parsing or the h-index / g-index / i10-index calculations) rather than to depend on ScholarFolio at runtime. The underlying public data it draws on is reachable directly through ORCID, OpenAlex, and Crossref.

## Privacy and reputational considerations

- The tool processes real people's professional information. With the owner-funded model, that text passes from the browser to the Worker to OpenRouter to the selected model provider. The Worker retains nothing beyond rate-limit counters, but the interface should state plainly which provider the text reaches and that provider's data policy. Mistral's EU hosting is an advantage for residency; DeepSeek is a Chinese provider, so routing identifiable researcher data through it warrants checking residency and training-use terms.
- A roast hosted on or near a professional domain can be misread without context. It should be kept off the index path, marked `noindex`, and framed clearly as deliberate, self-aware comedy.

## Resolved decisions

- Build path: a fresh repository from the template, built via Claude Code prompts (not a fork, not a single hand-written page).
- Key handling for the first version: a single owner key held in a Cloudflare Worker; no user-supplied keys.
- Spending control: per-IP rate limit in the Worker plus an OpenRouter account spend cap.
- Input for the first version: a single paste field with helper lines; automated retrieval deferred.
- Provider: OpenRouter, called from the Worker.

## Open decisions

- Default model slug (a flash-class model; confirm a current slug at build).
- Rate-limit values (roasts per IP per day) and the spend-cap figure.
- Roast prompt design and tone calibration.
- Whether any automated source (ORCID first) is added in the first version or held for later.
- Whether to ship the no-LLM template fallback as an offline mode.
- Whether to stream the output (likely yes) and how to present it.
